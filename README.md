

# Описание проекта
## Цель проекта

Создать инструмент для автоматизированной обработки и анализа результатов опроса сотрудников, в рамках которого оцениваются руководители трёх типов: непосредственные, вышестоящие и функциональные. Система позволяет структурировать ответы, выделить ключевые метрики и представить итоговую статистику в формате, удобном для дальнейшей работы с персоналом.

## Структура опроса
Анкета содержит 22 вопроса, объединённых в три блока в зависимости от того, какой руководитель оценивается:

- Непосредственный руководитель — руководитель по штатной структуре

- Вышестоящий руководитель — руководитель следующего уровня

- Функциональный руководитель — руководитель, не входящий в прямую иерархию, но ставящий задачи (технический руководитель, руководитель проекта и т.п.)
### Типы вопросов

| Тип | Описание | Формат ответа | Обработка |
|-----|----------|---------------|-----------|
| **Name** | Идентификация оцениваемого руководителя | Текст (ФИО) | Приведение к стандартному виду, сопоставление со справочником имён |
| **Check** | Фильтрующий вопрос о наличии руководителя | Да / Нет | Определяет, нужно ли обрабатывать последующие вопросы |
| **Select** | Оценочный вопрос с выбором варианта | Предопределённый список | Сводится к бинарной оценке (позитив / негатив) |
| **Number** | Числовая оценка | Целое число от 1 до 10 | Вычисляется среднее арифметическое |
| **Free** | Открытый комментарий | Произвольный текст | Агрегируется и сжимается с помощью YandexGPT |

## Что делает система
1. Идентификация руководителей
    - Приводит введённые имена к стандартизированному виду
    - Распознаёт инициалы, сокращения, разговорные формы
    - Сопоставляет с заранее подготовленным списком сотрудников
    - При необходимости обращается к YandexGPT для нормализации имён

2. Сбор статистики
    - Для каждого руководителя агрегируются ответы по всем вопросам
    - Подсчитываются доли позитивных ответов в вопросах типа Select
    - Вычисляются средние оценки для числовых вопросов
    - Собираются свободные комментарии

3. Анализ свободных ответов
    - С помощью YandexGPT производится сжатие множества отзывов до краткой сводки
    - Выделяются ключевые предложения по улучшению работы
    - Исключается эмоциональная окраска, остаётся суть

4. Формирование отчётов
    - Excel-отчёт — сводная таблица со статистикой по всем руководителям
    - PowerPoint-презентация — персонализированные слайды для каждого руководителя (по шаблону)
    - Интерактивные графики — визуализация распределения оценок

## Результаты, которые даёт система

| Результат | Формат | Описание |
|-----------|--------|----------|
| **Общая статистика по всем руководителям** | Excel | Сводная таблица с долями позитивных ответов, средними оценками и количеством отзывов для каждого руководителя по каждому типу вопросов |
| **Персональный отчёт руководителя** | PowerPoint | Индивидуальные слайды с оценками от разных типов подчинённых, выделением сильных сторон и зон роста, а также сжатой обратной связью от сотрудников |

## Масштабируемость и конфигурируемость

### JSON-описание структуры опроса

Вся логика обработки опроса определяется через JSON-конфигурацию. Это позволяет адаптировать систему под любой формат анкеты без изменения кода.

**Структура конфигурации:**
- **`fields`** — описание каждого вопроса: тип, формулировка, возможные варианты ответов (для select-вопросов)
- **`groups`** — группировка вопросов по типам руководителей (непосредственный, вышестоящий, функциональный)
- **`merge`** — объединение числовых вопросов из разных групп для расчёта агрегированных метрик
- **`output headers`** — кастомизация названий колонок в итоговых отчётах

### Поддержка новых типов ответов

Архитектура предусматривает расширение через **mini-collectors** — отдельные классы для каждого типа вопросов:

| Collector | Назначение |
|-----------|------------|
| `NumberCollector` | Сбор числовых оценок, расчёт среднего |
| `SelectCollector` | Сбор бинарных оценок, расчёт доли позитивных ответов |
| `FreeCollector` | Сбор текстовых комментариев, AI-сжатие |

Для добавления нового типа вопросов достаточно создать новый collector, реализующий методы:
- `add_info()` — обработка одного ответа
- `get_columns_names()` — формирование заголовков в отчёте
- `get_columns_values()` — формирование итоговых значений

### Преимущества подхода

- **Гибкость**: Изменение структуры опроса не требует переписывания кода
- **Расширяемость**: Добавление новых типов вопросов через новые collectors

# Необходимые структуры
Для обработки таблицы неплохо было бы создать JSON с описанием полей, и групп:
```
{
    "fields" : {
        "номер поля 1" : [
            "тип поля 1", 
            "имя поля 1", 
            {
                "ответ 1" : "интерпретация 1", 
                "ответ 2" : интерпретация 2"
            }
        ],
        "номер поля 2" : [
            "тип поля 2", 
            "имя поля 2"
        ],
        ...
        "номер поля n" : [
            "тип поля n", 
            "имя поля n", 
            {
                "ответ 1" : "интерпретация 1", 
                "ответ 2" : интерпретация 2"
            }
        ]
    },

    "groups" : {
        "имя группы 1" : [
            1,
            2,
            5,
            ...
            4
        ],
        ...
        "имя группы n" : [
            17,
            19,
            3,
            ...
            6
        ]
    }

    "merge" : {
        "результирующее поле 1" : [
            1,
            17,
            ...
            9
        ]
        ...
    }
}
```

Для удобства мы нумнеруем вопросы, чтобы дальше использовать их номера, а не их полную формулировку. Фактически мы создаём словарь, где ключом будет номер, а значением - список параметров. Для каждого номера мы определяем тип и формулировку вопроса. После этого могут идти дополнительная информация. У нас такая информация необходима для вопросов с выбором ответа и галочек.    
 

Для классификации полей можно использовать следующие типы:
- ```check``` - галочка, которая говорит о том, следует ли обрабатывать вопросы этой группы (опционально - ограничить одной на группу)
    - в дополнительной информации будет словарь ответов, где по формулировка ответа будет ключом, а значением - 0 или 1.     
- ```name``` - имя оцениваемого человека (опционально - ограничить одним на группу)
- ```select``` - вопрос с выбором ответа
    - в дополнительной информации будет словарь ответов, где по формулировка ответа будет ключом, а интерпретация - значением.     
- ```number``` - вопрос со вводом числа
- ```free``` - вопрос со свободным ответом т.е. обратная связь

Поскольку части таблицы могут собирать статистику для разных людей, то имеет смысл разделить обработку этих частей. Для этого создадим словарь ```groups```, где каждой именованной группе будем сопоставлять список полей этой группы. Так как группа должна быть объединена на основе одного ключевого поля, то имеет смысл проверять группу на корректность. Для галочек такое ограничение не обязательно.

Поскольку при объединении групп в общую таблицу может понадобиться общая статистика для полей из разных групп, то можно добавить словарь ```merge```, в котором будет указываться название нового поля и то, какие поля оно объединяет.  

Первоначально для каждой группы можно сделать две таблички - отчётную и со строками, в которых не удалось идентифицировать имена (опционально - дополнение отчётной таблички после правки необработанных имён).  

Для хранения промежуточных результатов можно использовать следующую структуру (назовём её ```collector```):    
```
{
    "name 1" : {
        "question 1" : mini_collector< T1 >,
        "question 2" : mini_collector< T2 >,
        ...
        "question n" : mini_collector< T1 >
    },
    ...
    "name m" : {
        "question 1" : mini_collector< T4 >,
        "question 2" : mini_collector< T4 >,
        ...
        "question n" : mini_collector< T5 >
    }
}
```

Формат хранения ответов на вопросы в ```mini_collector< T >``` будет зависеть от его типа ```T```:
- ```select``` - счётчики по вариантам ответов и общий счётчик 
    - в данном случае под вариантами ответов будут подразумеваться те ответы, которые получаются после расшифровки т.е. в нашем случае позитивная оценка/негативная оценка
- ```number``` - сумма по всем ответам и общий счётчик
- ```free``` - полный список сырых ответов на вопрос

Для типа ```name``` можно создать некоторую структуру ```name_dict```, которая будет являться словарём, в котором по набору токенов, на которые разбивается входная строка, будет выдаваться строка с именем. Все имена будут храниться в каком-то файле. Он может быть получен либо в результате анализа всех полей типа ```name```, либо подготовлен заранее.

# Процесс обработки

Предварительно имеет смысл проверить JSON на корректность структуры по части ```groups``` и ```merge```.

1. Читаем табличку в ```DataFrame```
2. Создаём общие ```name_dict``` и  ```survey_structure```
3. Разбиваем ```DataFrame``` на группы столбцов в соответствии с типом руководства, и для каждой группы создаём ```collector```, в котором будем собирать данные по столбцам
    - По списку полей в группе создаётся ```survey_substructure```, в которой будет храниться необходимая часть полей для данной группы
    - Определяются номера полей типа ```name``` и ```check```  
    - Переупорядочиваются поля так, чтобы первыми были ```check```, зачем ```name```, а потом остальные
    - Построчно анализируются идентифицированные персоналии, те кого не удалось идентиифицировать добавляются в соответствующую таблицу
4. Объединение ```collector``` в общий ```DataFrame``` 
    - ```collector```'ы различных групп объединяются в общий, в который добавляются поля из ```merge```
    - формируем ```DataFrame``` по столбцам
    - если какой-то столбец преобразуется в несколько, то добавляем стандартные для типа постфиксы
5. Из итогового ```DataFrame``` собираются ответы на "Общий отчёт"
